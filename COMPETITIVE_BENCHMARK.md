### 2.3 算力单位说明

**FPGA LPU 的 TMACs ≠ GPU 的 TFLOPS**，原因如下：

| 维度 | GPU Tensor Core | FPGA DSP |
|------|:---|:---|
| **乘法类型** | FMA (a×b+c)，计为 2 Ops | MAC (a×b+c)，计为 1 Op |
| **精度** | FP8×FP8→FP32 累加 | fp4×fp8→Q12 累加 (专有) |
| **利用率** | 60-80% (大模型) | 85-95% (固定数据流) |
| **有效算力折算** | 1 TFLOPS ≈ 0.5 TMACs | 1 TMAC ≈ 2 TFLOPS (GPU等价) |

**归一化方法**: 本报告使用「有效 Ops/s」统一衡量——即考虑精度、利用率和FMA折算后的实际乘加吞吐。

### 3.2 TTFT（Time-To-First-Token，首Token延迟）

FPGA LPU 采用 **Flash 模型 Prefill (双路 CPU 主力 + GPU 备选) + FPGA Full Decode** 异构架构：
- Prefill: DeepSeek V4 Flash (285B, 27 layers) — 仅 44% Full 模型的 MACs
- Chunked Prefill: 128 tokens/chunk, 用户看到首 token 仅需等第一个 chunk
- Primary: 双路 AMD EPYC 9755 (32 TFLOPS) → **首 token ~112ms**
- Fallback: L20 GPU → ~40ms (极少需要, 仅 sub-100ms SLA 场景)

| Prompt长度 | FPGA LPU (双路EPYC, chunk=128) | H200 | B300 | 950PR |
|------|:---:|:---:|:---:|:---:|
| **P=128** | **~112 ms** (1 chunk) | ~25 ms | ~15 ms | ~40 ms |
| **P=512** | **~112 ms** first / ~0.5s total | ~120 ms | ~65 ms | ~160 ms |
| **P=2048** | **~112 ms** first / ~2.0s total | ~500 ms | ~250 ms | ~650 ms |
| **P=8192** | **~112 ms** first / ~8.0s total | ~2.0 s | ~1.0 s | ~2.6 s |

> **首 token 延迟与 GPU 同级别 (~112ms vs H200 120ms)** — Chunked Prefill 是关键。
> 双路 EPYC 9755: 32 TFLOPS FP8 + 1.2 TB/s 内存带宽。
> GPU fallback 仅用于需要 <100ms 首 token 的极致低延迟场景。

### 3.4 KV Cache 容量与并发会话

DeepSeek V4 Pro 采用 MLA（Multi-head Latent Attention），KV Cache 存储压缩潜在表示：
- **K_latent**: 512 维 + RoPE 64 维 = 576 bytes/token/layer (FP8)
- **V_latent**: 512 维 = 512 bytes/token/layer (FP8)
- **每 token 每层**: 1,152 bytes (K+V, FP8)
- **每 token 全 61 层**: 70,272 bytes ≈ **68.6 KB**

KV Cache 按层分布在 32 芯片上（每芯片 ~2 层），每芯片 HBM 可用 31.3 GB（扣除权重后的 KV 专用空间）。

| 指标 | FPGA LPU | H200 (8-GPU) | B300 (8-GPU) | 950PR (8-NPU) |
|------|:---:|:---:|:---:|:---:|
| **每 token 全层 KV** | 68.6 KB (MLA FP8) | ~213 KB (FP8) | ~211 KB (FP8) | ~205 KB (FP8) |
| **节点 KV-HBM 总量** | 1,001 GB | ~800 GB | ~1,800 GB | ~700 GB |
| **MLA KV 密度优势** | **3.1× vs GPU** | 1.0× (基准) | 1.0× | 1.0× |

#### 不同上下文长度的并发会话数（HBM 容量限制）

| Context 长度 | FPGA LPU | H200 | B300 | 950PR | 典型场景 |
|------|:---:|:---:|:---:|:---:|------|
| **128K** | **~117** | ~35 | ~80 | ~40 | 长文档 QA |
| **256K** | **~58** | ~18 | ~40 | ~20 | 代码库分析 |
| **512K** | **~29** | ~9 | ~20 | ~10 | 大型代码库/合同 |
| **1M** | **~15** | ~5 | ~10 | ~5 | Agent 工作流 |
| **2M** | **~7** | ~2 | ~5 | ~2 | 长周期 Agent |

> **Agent 时代的核心矛盾**: 上下文长度每翻倍，并发能力减半。在 1M+ token 的 Agent 场景下，
> KV Cache 容量成为所有方案的首要瓶颈。FPGA LPU 因 MLA 原生压缩实现（68.6 KB/tok vs GPU 213 KB/tok）
> 拥有约 **3.1× KV 密度优势**，但绝对并发数仍受 HBM 物理容量制约。
>
> **缓解路径**: (1) KV Cache offload 到主机 RAM（PCIe 64 GB/s 带宽足够支撑 decode 时的
> KV 读取），(2) 滑动窗口注意力（SLIDING_WINDOW=128），(3) 跨层 KV 共享。

#### 瓶颈分析

```
会话并发瓶颈来源 (256K context):

FPGA LPU:  HBM容量(58) << 流水线吞吐(~1,744)
H200:      HBM容量(18) ≈  流水线吞吐(~15)
B300:      HBM容量(40) ≈  流水线吞吐(~35)

→ FPGA LPU 是 HBM 容量瓶颈，GPU 是 HBM+吞吐双瓶颈
```

### 4.3 能效来源分析

FPGA LPU 的极端能效优势来自三个架构决策：

1. **无指令开销**: 数据流固定流水线，无取指/译码/发射功耗（GPU SM 约 20% 功耗用于控制）
2. **定宽数据路径**: fp4×fp8 专用 MAC，无精度转换浪费（GPU Tensor Core 需支持 FP64→INT4）
3. **无冗余通信**: 12 专家/芯片固定分配，无 All-to-All 网络功耗（GPU MoE 需 NVLink 全互联）

## 6. Prefill / Decode 分解：为什么这不是妥协而是最优解

### 5.5 架构演进: FPGA-Only → CPU-Attention + FPGA-FFN

基于 MLA 计算量分析（Attention 仅占 0.02%，FFN 占 96%），架构从 v1 演进到 v2:

| | v1: FPGA Only | v2: CPU+FPGA | 变化 |
|------|:---:|:---:|:---:|
| Attention | FPGA | **CPU (AMX/SVE2)** | FPGA 去掉 attention |
| KV Cache | HBM 32GB | **DDR5 512GB** | 容量 16× |
| Expert FFN | FPGA | **FPGA** | 不变 |
| PCIe 开销 | 仅 Prefill KV | 14KB×61×2/层 | +30 GB/s |
| FPGA 资源 | Attn+KV+FFN | **FFN only** | 省 30% BRAM |
| @1M 会话 | ~15 | **~250** | 17× |

> 关键认知: MLA 把 attention 压到可忽略（131K MACs vs 462M FFN），CPU 做 attention 几乎零开销。
> FPGA 专注 Expert FFN，所有 DSP 用于 FFN 并行。KV 留在 DDR5，容量问题消失。

### 6.0 两个阶段的物理本质

Transformer 推理的 Prefill 和 Decode 是两个**物理特性截然不同**的计算阶段：

| 维度 | Prefill | Decode |
|------|:---|:---|
| **计算复杂度** | O(P²) 注意力 + O(P) FFN | O(P) 注意力 + O(1) FFN |
| **批量特性** | 大批量 (P=128~8192 tokens 并行) | 逐 token 串行 (batch=1~32) |
| **瓶颈类型** | **算力 + HBM 带宽** | **HBM 容量 + 延迟** |
| **最优并行模型** | SIMT 大规模并行 (GPU) | 固定流水线确定性执行 (FPGA) |
| **DSP/SM 利用率** | GPU 60-80% | FPGA 85-95%, GPU <10% |
| **指令开销** | 平摊到 P tokens, 可忽略 | 每 token 走完整 kernel 流程, 占比高 |
| **内存访问模式** | 连续大块读取权重 | 随机小块读取 KV cache |

**结论**: 不存在一种硬件能同时最优处理 Prefill 和 Decode。GPU 在 Prefill 上占优（Tensor Core + 8 TB/s HBM），FPGA 在 Decode 上占优（0 指令开销 + 确定性能延迟）。将两者分开是**物理上的最优分解，而非技术妥协**。

### 6.1 为什么 FPGA LPU 不适合 Prefill

```
Prefill P=512 的注意力计算量: P² × head_dim × num_heads ≈ 17B MACs/layer
→ 需要灵活的大矩阵乘法引擎
→ GPU Tensor Core: 原生 warp-level GEMM, 8 TB/s HBM 带宽喂饱
→ FPGA DSP: 12,300 个定点 MAC, 920 GB/s HBM → 延迟高 40×
```

### 6.2 为什么 GPU 不适合 Decode

```
Decode B=1 的计算量: P × head_dim × num_heads ≈ 33M MACs/layer
→ GPU SM 利用率 <10%, 其余 Tensor Core 全部闲置
→ CUDA kernel 启动延迟 ~5-10μs, 占每层延迟的 20-40%
→ FPGA: 确定性流水线, 0 指令开销, 每层 25μs
```

### 6.3 统一度量：token/kWh 为什么是正确指标

在 Agent 时代 (1M+ context)，**HBM 容量是硬上限**。`token/kWh` 同时衡量：
- **容量效率**: 每瓦特支撑多少 context（HBM 容量 ÷ 功耗）
- **计算效率**: 每瓦特产出的 decode token（TPS ÷ 功耗）

B300 的真正代际提升不在 FP4 算力（15 PFLOPS），而在 **288 GB HBM3e**（2× H200 的 141 GB）—这直接翻译为 2× 并发会话。算力提升是次要的。

```
token/kWh 归一化分解 (以 H200 = 1.0× 为基准):

FPGA LPU: 16.5× = KV密度(3.1×) × 利用率(1.3×) × 1/功耗(1.9×) × 架构(2.2×)
          └─ 3.1×: MLA FP8 原生 68.6KB/tok vs GPU 213KB/tok
          └─ 1.3×: DSP 90% vs SM 70% 利用率
          └─ 1.9×: 5.3kW vs 10kW
          └─ 2.2×: 0指令开销 + 确定性流水线 + 无All-to-All

B300:     1.6× = KV密度(1.0×) × 利用率(1.1×) × 1/功耗(0.7×) × 架构(2.1×)
          └─ 2.1×: FP4精度叠加 + TensorRT-LLM软件优化
          └─ 0.7×: 14kW vs 10kW (功耗倒退, 被HBM翻倍抵消)

950PR:    1.2× = KV密度(1.0×) × 利用率(0.9×) × 1/功耗(1.7×) × 架构(0.8×)
          └─ 1.7×: 6kW vs 10kW (低功耗优势)
          └─ 0.8×: CANN软件栈效率损耗 (<1.0)
```

**一眼看出结论**:
- **FPGA LPU 的 16.5× 来自四个因子全 >1.0**——每个维度都赢
- **B300 靠架构因子 2.1× 弥补功耗倒退**——HBM 翻倍被功耗翻倍吃掉，净提升靠 FP4
- **950PR 靠低功耗撑着**——但利用率(0.9×)和软件栈(0.8×)都是拖累
- **归一化到 token/kWh 后，排名非常干净**: FPGA LPU (16.5×) >> B300 (1.6×) > 950PR (1.2×) > H200 (1.0×)

## 7. 架构路线对比

### 7.1 Decode-Only vs 训推一体

| 维度 | FPGA LPU | GPU (H200/B300) | NPU (昇腾/寒武纪) |
|------|:---|:---|:---|
| **架构范式** | 空间计算（固定数据流） | SIMT/SIMD 通用计算 | SIMD + 专用矩阵引擎 |
| **Pre训练支持** | ❌ 不支持 | ✅ 完整支持 | ✅ 部分支持 |
| **Fine-tuning** | ❌ 不支持 | ✅ 完整支持 | ✅ 部分支持 |
| **推理-Decode** | ✅ **核心优势** | ✅ 支持 | ✅ 支持 |
| **推理-Prefill** | ⚠️ CPU协处理 | ✅ GPU原生 | ✅ NPU原生 |
| **模型灵活性** | ❌ 单一模型固化 | ✅ 任意模型 | ⚠️ 框架适配 |
| **编译器依赖** | Quartus (一次综合) | TensorRT-LLM (运行时) | CANN/TensorFlow |
| **软件栈成熟度** | 低 (自研) | 极高 (CUDA) | 中 (CANN) |

### 6.2 FPGA LPU 架构优势

```
传统 GPU 推理路径:
  Token → 取指令 → 译码 → 发射 → 读寄存器 → TensorCore → 写寄存器 → 下一层
          (每层数百条指令，分支预测、缓存未命中)

FPGA LPU 推理路径:
  Token → DSP Pipeline (Attn→RMS→Router→FFN→RMS) → 下一层
          (0条指令，完全流水线化，确定性能延迟)
```

| 优势 | 量化效果 |
|------|:---|
| **零指令开销** | GPU 20-30% 功耗用于控制逻辑 → FPGA 全部用于计算 |
| **确定性能延迟** | 每层 ~25 μs，无抖动 → SLA 可精确保证 |
| **固定专家分配** | 12 专家/芯片，无 All-to-All → 跨芯片通信减少 96% |
| **专用精度路径** | fp4×fp8→Q12 全链路定制 → 无精度浪费 |
| **KV Cache 效率** | MLA 576B/token vs 标准 1152B → 容量翻倍 |

### 6.3 FPGA LPU 架构劣势

| 劣势 | 影响 | 缓解措施 |
|------|:---|:---|
| **仅支持单一模型** | 无法切换模型架构 | 面向 DS V4 Pro 专用部署，模型升级需重新综合 |
| **CPU 预填充瓶颈** | 长 prompt TTFT 偏高 | P0: FPGA 侧 fp4 预填充 (规划中) |
| **开发周期长** | RTL 修改→综合→布局布线 → 数小时 | 增量编译 + 预编译 IP 库 |
| **缺乏软件生态** | 无可用的推理框架/编译器 | 自研 vLLM 调度层 + Python 配置工具 |
| **无法训练/微调** | 仅推理 | 适合推理即服务 (IaaS) 场景 |
