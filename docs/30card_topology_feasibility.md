# DeepSeek V4 Pro — 30×Agilex 7 M 组网拓扑可行性方案

> 基于真实 config.json 架构参数，工程可落地。
> 纯 Ethernet 拓扑，零 PCIe Switch。

---

## 一、设计原则

**一张网，一个协议，双上联。**

```
所有 FPGA↔FPGA 通信统一走 F-Tile 200GbE, 经双 ToR 交换机
PCIe 仅用于 Host↔FPGA 控制面 (权重加载、寄存器访问、中断)
不设 PCIe Switch → 无 PCIe 单点 → 全网统一 Ethernet

为什么 Ethernet 延迟可接受:
  F-Tile 200GbE 延迟 ~1.5μs vs PCIe P2P ~260ns → 差距 ~1.2μs
  61 层 × 1 跳/层 × 1.2μs = 73μs → 占单 token 10ms 的 0.73%
  完全淹没在 HBM 权重加载 (~6ms) 中
```

## 四、通信带宽核算

所有 FPGA↔FPGA 通信统一经 Ethernet。原来区分"组内 PCIe P2P"和"跨组 F-Tile"的分类不再存在。

### 4.1 逐链路带宽需求

```
┌──────────────────────────────────────────────────────────────────┐
│                    每 FPGA 片间流量分解                            │
├──────────────────────────────┬──────────┬──────────┬─────────────┤
│ 通信类型                      │ 频率     │ 数据量    │ 带宽需求     │
├──────────────────────────────┼──────────┼──────────┼─────────────┤
│ TP All-Reduce (含节点内/跨)   │ 61 层    │ 224KB/层 │ 213 MB/s    │
│ MoE Dispatch (含节点内/跨)   │ 58 层    │ 变化     │ ~2,400 MB/s │
│ Pipeline 边界                 │ 2 边界   │ 7KB/边界 │ <1 MB/s     │
│ KV Cache 同步                 │ 每 token │ 小       │ <100 MB/s   │
├──────────────────────────────┼──────────┼──────────┼─────────────┤
│ 合计 (单向)                   │          │          │ ~2,700 MB/s │
│ 合计 (双向)                   │          │          │ ~5,400 MB/s │
└──────────────────────────────┴──────────┴──────────┴─────────────┘
```

### 4.2 链路利用率

```
单 FPGA → ToR 链路: 200GbE (25 GB/s 有效)

  单向需求: 2.7 GB/s
  利用率: 2.7 / 25 = 10.8%

  结论: 超过 9× 余量，节点内外的流量差异无影响。
```

### 4.3 延迟预算

```
单次 FPGA→FPGA 经 ToR (同节点或跨节点):
  FPGA F-Tile 组帧:       ~50 ns
  F-Tile MAC/PCS 发送:    ~200 ns
  ToR cut-through 转发:   ~300 ns
  F-Tile MAC/PCS 接收:    ~200 ns
  目标 FPGA 解帧:          ~50 ns
  ─────────────────────────────
  单跳总计:               ~800-1,500 ns (取决于 ToR 负载)

vs PCIe P2P (旧方案): ~260 ns
差距: ~1μs/跳

对整体推理延迟的影响:
  → 61 层每层最坏 1 跳 × 1μs = 61μs
  → 单 token decode 总延迟 ~10ms (HBM 权重加载占 ~6ms)
  → 影响: 61μs / 10ms = 0.61%
  → 可忽略
```

## 五、ToR 交换机选型

### 5.1 推荐配置

| 参数 | 规格 | 备注 |
|------|------|------|
| 数量 | 2 台 | active-active MLAG |
| 端口 | 32 × 400GbE QSFP-DD | 或 64 × 200GbE QSFP56 |
| 每 ToR FPGA 连接 | 30 × 200GbE (breakout) | Port0→ToR-A, Port1→ToR-B |
| 每 ToR Host 连接 | 4 × 25GbE | 每节点 Host 1 条 |
| 交换机间互联 | 2 × 400GbE (MLAG peer link) | |
| RoCE 特性 | PFC / ECN / DCQCN | 无损以太网 |
| 参考型号 | NVIDIA Spectrum-3 SN4600C 或兼容 | 32×400G QSFP-DD |

### 5.2 端口规划 (每台 ToR)

```
ToR-A (32 × 400GbE QSFP-DD):

  Ports 00-14: 30 × 200GbE → FPGA00-29 F-Tile Port0
    (400GbE breakout 2×200GbE, 用 15 个物理口)

  Ports 15-18: 4 × 25GbE → Node0-3 Host 主板

  Ports 19-20: 2 × 400GbE → ToR-B (MLAG peer link)

  Ports 21-31: 预留 (上行/扩容)

ToR-B: 镜像对称
```

## 六、单节点物理实现

```
标准 4U GPU 服务器 (如 Supermicro SYS-821GE):

┌──────────────────────────────────────────────────┐
│  Dual Intel Xeon (Sapphire Rapids)               │
│  ┌───────────┐        ┌───────────┐             │
│  │  CPU 0    │◄─UPI──►│  CPU 1    │             │
│  │  PCIe ×4  │        │  PCIe ×4  │             │
│  └──┬──┬──┬──┘        └──┬──┬──┬──┘             │
│     x8 x8 x8 x8         x8 x8 x8 x8             │
│     │  │  │  │          │  │  │  │              │
│  ┌──┴──┴──┴──┴──────────┴──┴──┴──┴──┐           │
│  │          FPGA ×8                  │           │
│  │    (Slot 0-7, PCIe x8 直连 CPU)   │           │
│  │     每卡 QSFP-DD Cage ×2          │           │
│  └────────────────┬──────────────────┘           │
│                   │ QSFP-DD ×16                  │
│                   │ (每卡 2×200GbE → 双 ToR)     │
│                   │                              │
│  主板 25GbE ──────┼──→ ToR (控制面)              │
└───────────────────┼──────────────────────────────┘
                    ▼
              ToR-A + ToR-B
```

**关键：无 PCIe Switch。** 8 FPGA 的 PCIe x8 直连双 CPU 的 Root Port。
Sapphire Rapids 单 CPU 提供 80 条 PCIe 5.0 Lane，4 个 x8 = 32 Lane，双 CPU 共 64 Lane 即可覆盖 8 卡。
标准 GPU 服务器主板 (如 SYS-821GE) 的 8 个 x16 插槽物理上已均衡分布到双 CPU。

PCIe 仅用于 Host↔FPGA：
- 权重首次加载 (32 GB / 28 GB/s ≈ 1.1s)
- 寄存器 MMIO 访问 (推理指令下发，<1μs)
- MSI-X 中断 (推理完成 / 异常，<1μs)
- 固件升级 / Signal Tap 远程抓取

数据面完全不走 PCIe。

## 七、容错设计

### 7.1 四类故障覆盖

```
┌──────────────────┬────────────┬──────────────┬──────────────┐
│ 故障              │ 检测        │ 恢复          │ 吞吐影响      │
├──────────────────┼────────────┼──────────────┼──────────────┤
│ 单 FPGA (①)      │ <30ms      │ 热备接管 <500ms│ 无           │
│ 单 F-Tile 端口 (②)│ <1μs       │ Port 切换 <1ms │ 无           │
│ 单 ToR (③)       │ <1μs       │ 全局切 <1ms    │ 无           │
│ Host (④)         │ <100ms     │ 邻 Host 接管   │ 无           │
└──────────────────┴────────────┴──────────────┴──────────────┘
```

### 7.2 单卡故障切换

```
热备:
  Node 0: 7 活跃 + 1 热备
  Node 3: 7 活跃 + 1 热备

热备卡预加载本节点全部活跃卡权重 (~2.4-4.7 GB, 常驻 HBM)
双 F-Tile 端口均已 link up, RoCE QP 预建立

切换:
  T+0:     调度器确认故障
  T+50ms:  激活热备卡 (寄存器 + 路由表全集群广播)
  T+100ms: KV Cache 重建 (RoCE Read 邻卡, ~2.2MB × 4 / 25GB/s ≈ 0.4ms)
  T+500ms: 恢复推理
```

### 7.3 单 ToR 故障

```
所有 30 张 FPGA 的对应 F-Tile 端口同时 link down (<1μs)
→ RoCE multipath 自动切换, 全部流量走另一 ToR
→ <1ms, 无丢包

单台 ToR 总带宽余量:
  30 FPGA × 2.7 GB/s = 81 GB/s 总需求
  30 × 200GbE = 750 GB/s 总上行 (单 ToR)
  利用率: 10.8% → 有 9× 余量
```

### 7.4 可用性

```
单卡 MTBF ~50,000h, 30 卡系统级 MTBF ≈ 1,667h (2.3 月)
2 备卡 → 连续两次故障 (1 周内) 概率 ~1%

ToR MTBF > 200,000h (23 年), 双 ToR 同时故障 < 0.2%

系统可用性 > 99.97%
```

## 九、结论

**30 卡纯 Ethernet 拓扑在工程上完全可行，且比 PCIe+P2P 方案更简洁可靠。**

- **拓扑**：双 400GbE ToR Switch (MLAG)，30 FPGA 双上联，零 PCIe Switch
- **每 FPGA 链路**：2 × 200GbE F-Tile → ToR-A + ToR-B
- **带宽利用率**：10.8%（单链路），9× 余量
- **延迟**：Ethernet ~1.5μs/跳 vs PCIe P2P ~260ns，差距仅占单 token 延迟 0.6%
- **容错**：4 类故障全自动恢复 (<500ms)，可用性 > 99.97%
- **复杂度**：单平面 vs 旧方案双平面，故障模式从 6 种降到 4 种
