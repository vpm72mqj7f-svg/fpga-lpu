### Step 1.6: Open a Waveform and Identify the MAC Pipeline

The simulation generated a VCD (Value Change Dump) file. Open it:

```bash
gtkwave tb_fp4_mac.vcd
```

In GTKWave, find these signals in the SST hierarchy (left panel, expand `tb_fp4_mac`):

| Signal | Description | What to look for |
|--------|-------------|-----------------|
| `clk` | 100 MHz clock (bring-up) | Rising edge every 5 ns |
| `rst_n` | Active-low reset | Low for first 2 cycles, then high |
| `mac_in.weight[3:0]` | 4-bit fp4 weight input | Changes to `4'h4` (=1.0) during test 1 |
| `mac_in.activ[7:0]` | 8-bit fp8 activation input | `8'h38` (=1.0) during test 1 |
| `mac_in.scale[11:0]` | Pre-decoded 12-bit scale | `12'h100` (=256, 1.0 x 256) |
| `dut.s0_weight[3:0]` | Pipeline stage 0 (input register) | Weight captured on valid cycle |
| `dut.s1_w_signed` | Stage 1 (decode, sign-extend) | Decoded into magnitude + sign |
| `dut.s2_base_product` | Stage 2 (base multiply, DSP) | fp4 x fp8 multiply result |
| `dut.s3_product` | Stage 3 (scale multiply, DSP) | Base product x scale factor |
| `dut.accumulator[31:0]` | Stage 4 (accumulate) | Running sum, saturates on overflow |
| `mac_out.result[31:0]` | FP32 output | `32'h3f800000` (= 1.0) |
| `mac_out.valid` | Output valid strobe | Pulses 1 cycle when result is ready |

**Exercise:** Find the 6-cycle delay between `mac_in.valid` going high and `mac_out.valid` going high. The pipeline is: Stage 0 (input reg) -> Stage 1 (decode) -> Stage 2 (base multiply) -> Stage 3 (scale multiply) -> Stage 4 (accumulate) -> Output register. 5 stages + 1 output reg = 6 cycles latency.

**Exercise 2:** Find where weight `4'h4` appears at `mac_in.weight`, then at `dut.s0_weight` (1 cycle later), then at `dut.s1_w_signed` (2 cycles later). This is the pipeline moving data forward each clock.

Read the ASCII architecture diagram:

  -> Layer 00: MLA Attention (SRAM weights, DSP compute)
  -> Layer 00: MoE Router selects top-6 experts from 384

- Per-token, per-layer compute: ~611M MAC
- Single token B=1 decode latency: ~1,510 us (full 61 layers)
- Per-layer average: ~24.7 us
- C2C pipeline forward: ~250 ns per hop
- 32 tokens in-flight simultaneously (one per chip at steady state)

### Step 2.5: Quiz Yourself

Answer these five questions without looking at the docs. If you cannot answer all five, re-read the relevant section.

**Q1: What happens to a token that has 1 local expert vs 0 local experts vs 2+ local experts?**

- **0 local experts (82.7% probability):** All 6 selected experts are on other chips. SRAM-only compute (attention + shared expert + router + RMSNorm). HBM is completely idle. MoE dispatch sends the activation to the remote chips via C2C; MoE reduce collects results. This is the common case, and the architecture is optimized for it.

- **1 local expert (16.5%):** One expert lives on this chip. HBM streams that expert's fp4 weights (~33 MB) into the systolic array. The other 5 experts are computed remotely. Slight HBM bandwidth hit but still dominated by DSP compute.

- **2+ local experts (0.8%):** Rare. Two or more experts on the same chip. Both stream from HBM, potentially competing for bandwidth. Latency is marginally higher because two HBM weight loads must serialize on the same HBM controller.

**Q2: Why are attention weights in SRAM rather than HBM?**

Because they are needed every layer, deterministically. Loading attention weights from HBM every layer would consume ~10.1 us per layer of HBM bandwidth, reducing throughput ~40%. SRAM residency means HBM is idle during compute in 81.6% of layers (the zero-local-hit case).

**Q3: What is the difference between `top_master.sv` and `top_slave.sv`?**

`top_master.sv` is for Chip 0 of each card. It has PCIe 5.0 R-Tile hard IP (host interface). `top_slave.sv` is for Chips 1-3 of each card. Same compute RTL (`chip_top.sv`), same HBM2e, same C2C ports. No PCIe IP -- saves ~15-20% logic area. The `IS_PCIE_MASTER` parameter gates the PCIe instantiation.

**Q4: What is the function of `mla_attention_v2.sv`?**

It is the full MLA (Multi-head Latent Attention) pipeline: Q/K/V low-rank projection -> RoPE (rotary position embedding) -> KV cache write -> KV cache read -> attention score computation (Q*K^T) -> softmax -> output projection (A*V -> O). K and V are stored in compressed latent form (512-dim) in the KV cache, then decompressed on-the-fly during attention score computation. This saves 14x KV cache memory vs storing full expanded forms.

**Q5: What does `FPGA_LPU_PRODUCTION` control and when should you define it?**

It is a single `` `define `` in `lpu_config.svh` that switches between bring-up (HIDDEN=8, 30s sim) and production (HIDDEN=7168, 4-6h Quartus compile) parameter sets. Define it in Quartus `.qsf` files only: `set_global_assignment -name VERILOG_MACRO "FPGA_LPU_PRODUCTION"`. Never define it for Icarus simulation.

## Common First-Week Mistakes

### 1. Modifying `rtl/legacy/` Instead of the v2 Modules

**Symptom:** You edit `rtl/legacy/fp4_systolic_array.sv` or `rtl/legacy/mla_attention.sv`, and your change has no effect on production builds.

**Why:** Legacy modules (`rtl/legacy/`) are commented out in `hw/quartus/common/common_modules.qsf`. They exist for reference only. All production bitstreams use the v2 replacements.

**Fix:** Always work in the active module directories: `rtl/dsp/`, `rtl/attention/`, `rtl/moe/`, `rtl/chip/`, etc. If you are unsure which file is the "live" one, check `common_modules.qsf`:

```bash
cat hw/quartus/common/common_modules.qsf | grep -v "^#"
```

Any file on this list is active. Any file NOT on this list is legacy or not yet integrated.

---

### 2. Using Production Parameters in Icarus

**Symptom:** Icarus crashes with an out-of-memory error, or hangs for hours.

```
ERROR: Memory overflow at array allocation (7168 x 7168)
```

**Why:** You somehow defined `FPGA_LPU_PRODUCTION`, which switches all parameters to production size. A 7168x7168 matrix requires gigabytes of simulation memory.

**Fix:** NEVER pass `-DFPGA_LPU_PRODUCTION` to iverilog. The Makefile does not do this. If you are calling `iverilog` directly, do not add this define. If you edited `lpu_config.svh` to make production the default, revert your change.

```bash
# Quick sanity check:
grep "LPU_HIDDEN" rtl/include/lpu_config.svh
# Should show LPU_HIDDEN = 8 (the bring-up value is in the `else branch)
```

---

### 3. Forgetting to Regenerate Golden Vectors After Changing fp4 Arithmetic

**Symptom:** You change `fp4_mag_to_scaled()` LUT values, `fp8_to_scaled12()` decoding, or the accumulation logic. All simulation tests still PASS, but they pass because they are comparing against YOUR changed RTL output, not the Python golden reference.

**Why:** The self-checking testbench compares RTL output against pre-generated golden vectors. If you change the arithmetic, the golden vectors are now stale. The testbench is comparing your new (potentially wrong) RTL against old golden values, and silently diverging.

**Fix:** After ANY change to fp4 encode/decode/compute logic:
```bash
cd scripts/simulation
python gen_tb_vectors.py       # Regenerate golden vectors FROM Python reference
cd ../../rtl/sim
make SIM=iverilog               # Verify RTL now matches regenerated golden
```

If your change is intentional and the golden vectors need updating, this is correct. If your change accidentally broke the arithmetic, the regenerated golden vectors will now FAIL against your RTL -- which is exactly what you want to catch.

---

### 4. Not Checking Quartus Synthesis Warnings

**Symptom:** Your Icarus simulation passes, but the Quartus synthesis log contains warnings like:
```
Warning (10631): Inferred latch for signal "out_data"
Warning (10230): Width mismatch -- truncating 32-bit expression to 16 bits
```

**Why:** Icarus is lenient. Quartus is strict. An inferred latch in Quartus means your `always_comb` block is missing an `else` branch. Width mismatches mean you are silently truncating data. Both are bugs.

**Fix (inferred latch):** Every `always_comb` must have a complete assignment for every output in every branch:
```systemverilog
// WRONG: missing else -- infers latch
always_comb begin
    if (state == S_ACTIVE) out_data = in_data;
end

// CORRECT: always_ff for registered outputs
always_ff @(posedge clk) begin
    if (state == S_ACTIVE) out_data <= in_data;
end
```

**Fix (width mismatch):** Use explicit size casts:
```systemverilog
// Instead of: assign narrow = wide_signal;
assign narrow = wide_signal[NARROW_WIDTH-1:0];  // explicit truncation
```

---

### 5. Mixing Tabs and Spaces in SystemVerilog Files

**Symptom:** Inconsistent indentation when viewed in different editors. Git diffs show whitespace-only changes.

**Why:** Icarus and Quartus do not care, but your teammates do. The project convention is **spaces only** (4 spaces per indent).

**Fix:** Configure your editor:
```bash
# VSCode: add to settings.json
"editor.insertSpaces": true,
"editor.tabSize": 4,
"[systemverilog]": {
    "editor.insertSpaces": true,
    "editor.tabSize": 4
}
```

If you accidentally committed tabs, do a whitespace-only fixup commit before opening your PR.

---

### 6. Changing Interface Structs Without Updating All Consumers

**Symptom:** You add a field to `stream_beat_t` in `rtl/interfaces/avalon_stream.svh`. Your testbench compiles and passes. But 15 other testbenches now fail to compile because they use the old struct width.

**Why:** Interface structs (`avalon_stream.svh`, `c2c_packet.svh`, `pcie_dma.svh`) are included by every module that communicates across chip boundaries. Changing a struct changes the packed width, breaking all consumers.

**Fix:** After any interface change, run EVERY testbench, not just yours:
```bash
cd rtl/sim
# Compile and run each testbench to catch all consumers of the changed struct
```

If you must change an interface, coordinate with the other RTL engineers first. Interface changes are flagged in the daily RTL sync meeting.

---

### 7. Forgetting the `-g2012` Flag with Icarus

**Symptom:**
```
Error: syntax error near 'package'
Error: syntax error near 'typedef'
Error: syntax error near 'always_ff'
```

**Why:** Without `-g2012`, Icarus uses Verilog-2005 mode, which does not support SystemVerilog features (`package`, `typedef struct packed`, `always_ff`, `enum`, `logic`).

**Fix:** Always use `iverilog -g2012`. The Makefile includes this by default. If calling iverilog directly, always add it.
